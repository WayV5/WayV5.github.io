通用功能

统一返回对象R类

继承了HashMap类，封装了静态方法，最基础字段就是状态码code+信息msg，还能往里添加其他返回体信息。

后端验证

  • 虽然前端验证已经限制了用户请求中提交数据的规范性，但是为了防止有人通过http工具postman等，绕过浏览器直接发起http请求，后端验证还是必要的。

  • 项目中需要验证的requstbody形式的请求，封装好请求的表单form类，通过注解（javax.validation中的）对字段进行约束。

统一异常处理怎么做的？

  • 创建一个全局异常处理的配置类，用@RestControllerAdvice（@ControllerAdvice + @ResponseBody）注解修饰这个类，相当于给所有Controller类织入了一个AfterThrowing抛异常通知;

  • 然后这个异常处理类中的handler方法加@ExceptionHandler注解，对不同的异常类型，进行相应的异常处理。

用户模块

登录

密码加密

  • 采用AES加密，这是一种对称加密方式，对每个用户的密码加密时，密钥选择用户的用户名;
  • 即：以username为密钥给password加密，同样能解密；
  • sql有AES加解密的对应函数，操作方便

login

  • 先根据username和password去数据库查询userId，如果查询是Null，表示用户不存在，返回登录失败信息；

  • 如果查询到了，用权限框架StpUtil.setLoginId(userId)，生成token保存在redis，token是key，value是userId；

  • 然后将token返回给前端，因为cookie被禁用了，所以不能通过set-cookie的方式传递token，这里是放在了响应体里，前端解析出来放在localStorage/sessionStorage中，之后每次发起请求时带上token。

logout

后端：删除redis中的token，调用权限框架的方法就可以了;
前端：踢下线 -> 删除localStorage中的token和permissions，控制页面跳转（js代码）;

查询

用甘特图显示各个会议室的会议数据，所以要把一个会议室下的多条会议记录都合并起来；

  1. 把每条会议记录的各个字段，合并成jsonObject形式；
  2. 然后对同一个会议室place中的每个记录用group_concat合并起来；
  3. 最后还要将会议信息和会议室信息关联起来，再表关联一下。
  4. 其中还要涉及条件筛选，timestampdiff, concat等函数

新增（用到了异步线程）

  1. 业务逻辑层先在数据库中插入一条会议的数据，然后开启一个新的线程，异步调用工作流项目的方法，工作流项目会为这个会议创建一个工作流实例，并且返回instanceId，异步线程拿到返回值之后，再调用持久层方法，将instanceId字段更新到数据库中的会议记录中;

  2. 调用hutool包下的工具类提供的方法，发起http请求;
    post方法，请求方法的url ，把请求数据封在json里，转化成字符串传进请求体，然后execute();

  3. 得到resp，先判断以下响应码，然后拿出响应体，再执行业务逻辑。

为什么要异步？因为调用工作流项目的方法创建实例，工作流方法所需要的参数还要几个sql查询出来，再加上方法执行时间，如果同步调用客户端会等的时间比较久；并且这个业务客户端只要看见自己的请求成功了就行。

删除

​ 先删除数据库中的记录，然后用异步线程调用工作流项目的方法，删除会议对应的工作流实例

微信支付

商品订单（罚款单）：需要一个全局唯一的id，这里用的uuid；

支付订单：后端项目向微信平台请求得到的支付订单，也有一个全局唯一的Id

流程

  1. 生成支付订单；
  2. 用户拿手机扫码支付；
  3. 接收付款通知；（后端、支付的用户手机）
  4. 后端向前端推送付款消息；
  5. 提供主动查询付款状态的功能

支付订单生成

  • 前端项目ajax请求后端项目， 后端项目拿着罚款单，向微信支付平台发起http请求创建支付订单，后端拿到支付订单，生成QRcode，返回给前端。

  • 由于微信支付平台不信任前端（用户），因为不能识别用户信息，不能让用户随意创建支付订单；

  • 而后端项目请求支付订单，要携带者商家信息，做违法乱纪的事能查到。

后端系统调用微信支付api创建订单，需要验证企业商户身份，要上传工商登记证明；亲戚借的。。。

调用微信支付api，需要密钥和数字证书

代码实现

  1. 使用微信支付官方提供的工具类，创建支付订单；
  2. 拿出响应中的codeUrl，生成二维码，然后用base64进行编码，返回给前端；

用户支付

  • 用户，后端系统（商家系统）和支付平台，三方互相不信任；

  • 用户，看到了浏览器上的支付订单二维码，扫描，支付之前要向微信平台发起请求，查询订单信息，看一下金额，订单信息，防止后端系统申请支付订单时更改了支付金额；

  • 用户核对确认之后，就可以输入密码，扣款了；

  • 微信平台扣款之后，将通知发送给 后端系统（商户系统） 和 用户（扫二维码的手机）

后端接收付款结果

  1. 编写回调方法，用于微信平台通知付款结果；
  2. 微信平台付款通知的request中是XML格式的数据，通过字符流Reader读，是String格式的XML；通过工具类，将XML转换成map，便于提取数据；
  3. 通过工具类，验证一下数字证书的有效性，再验证一下通信状态码和业务状态码，没问题后更新数据库中的支付状态；
  4. 通过websocket向浏览器推送支付成功的消息；
  5. 后端还要在response中返回确认消息，表示后端接收到了支付成功通知；response通过writer字符流写入XML格式数据。

后端推送付款结果给前端

  1. 前端通过定时器，轮询的发起ajax请求，查询后端中订单是否支付成功，如果支付成功，则返回支付成功的响应，就隐藏支付二维码，弹出支付成功的提示消息。

    缺点就是给后端系统和数据库的负载增大。

  2. 使用websocket长连接，服务端将websocket连接缓存起来，服务端可以主动给浏览器推送消息。websocket设置过期时间，防止长时间占用网络带宽。

SpringBoot整合websocket

  • websocket是长连接协议，它发送请求是不能携带cookie或者LocalSession，前端要字节写代码，发送userId；

  • 后端也不能通过SaToken直接从redis拿出userId，要自己拿出userId；

  • springboot配置websocket：

    1. 写配置类，开启websocket；

    2. 声明websocket的生命周期函数，特别是onMessage()，接收前端消息的函数，要先判断消息类型，如果是”ping”操作，表示是一个心跳检测，用于维持websocket连接；如果是其他业务操作，则执行响应的业务操作。

    3. 前端发起websocket的请求时，后端会创建相应的session对象，后端要将session缓存起来，等想推送消息的时候，拿出来用；项目中用一个全局的静态Map来缓存session对象，由于是全局的，所以涉及到线程安全问题，因此采用的是ConcurrentHashMap。key是userId，value是session对象。

    4. onClose()方法，在连接关闭的时候，清除session的缓存。

    5. websocket连接有过期时间60s，过期自动销毁；要想保持连接，前端要定时发送心跳检测，以保持连接。

主动查询付款结果

​ 如果微信支付平台扣款成功了，但是可能网络原因，后端系统和用户手机都没收到通知，怎么办？

  这里用一个方法，浏览器点击查询支付状态按钮，发起ajax请求，后端系统主动向支付平台查询支付状态，如果支付成功了，支付平台就再发送一次通知；并且后端如果收到通知后，会刷新页面，用户就能看到支付结果了。

如果用户手机没弹出支付成功消息，但是后端系统收到了消息，并且已经在浏览器弹出了提醒，那么手机上也没必要再收到消息了。
